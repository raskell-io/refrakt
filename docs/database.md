# Database

Refrakt supports PostgreSQL and SQLite. The database choice is made at
project creation time and affects generated repos and migrations.

## Setup

### PostgreSQL

```bash
refrakt new my_app --db postgres
```

This adds `pog` to dependencies and generates a `repo.gleam` that
connects via URL:

```gleam
// src/<app>/data/repo.gleam
pub fn connect(cfg: Config) -> Result(pog.Connection, Nil) {
  let db_url = case cfg.env {
    config.Test -> "postgres://localhost:5432/my_app_test"
    _ -> "postgres://localhost:5432/my_app_dev"
  }

  let pool_name = process.new_name(prefix: "my_app_db")
  use db_config <- result.try(pog.url_config(pool_name, db_url))
  case pog.start(db_config) {
    Ok(_started) -> Ok(pog.named_connection(pool_name))
    Error(_) -> Error(Nil)
  }
}
```

Create the database:

```bash
createdb my_app_dev
createdb my_app_test
```

### SQLite

```bash
refrakt new my_app --db sqlite
```

This adds `sqlight` to dependencies and generates a `repo.gleam` with
a path helper:

```gleam
// src/<app>/data/repo.gleam
pub fn database_path(cfg: Config) -> String {
  case cfg.env {
    config.Test -> ":memory:"
    _ -> "my_app.db"
  }
}
```

No external database server needed. The SQLite file is created
automatically on first use.

## Repos

Repos are generated by `refrakt gen resource`. They use raw SQL
with typed decoders.

### PostgreSQL repos (Pog)

```gleam
pub fn list(db: pog.Connection) -> List(Post) {
  pog.query("SELECT id, title, body, published FROM posts ORDER BY id DESC")
  |> pog.returning(post_decoder())
  |> pog.execute(db)
  |> result.map(fn(r) { r.rows })
  |> result.unwrap([])
}

pub fn get(db: pog.Connection, id: Int) -> Result(Post, Nil) {
  pog.query("SELECT id, title, body, published FROM posts WHERE id = $1")
  |> pog.parameter(pog.int(id))
  |> pog.returning(post_decoder())
  |> pog.execute(db)
  |> result.replace_error(Nil)
  |> result.try(fn(r) {
    case r.rows { [item] -> Ok(item)  _ -> Error(Nil) }
  })
}
```

Postgres uses `$1`, `$2` parameter placeholders and `pog.parameter()`
for type-safe parameter binding.

### SQLite repos (Sqlight)

```gleam
pub fn list(db_path: String) -> List(Task) {
  use conn <- sqlight.with_connection(db_path)
  case sqlight.query(
    "SELECT id, title, completed FROM tasks ORDER BY id DESC",
    on: conn,
    with: [],
    expecting: task_decoder(),
  ) {
    Ok(rows) -> rows
    Error(_) -> []
  }
}
```

SQLite uses `?` parameter placeholders and `sqlight.int()`,
`sqlight.text()` for parameter binding. Boolean fields use
`sqlight.decode_bool()` since SQLite stores booleans as integers.

## Decoders

Both backends use positional decoders (field index, not name):

```gleam
fn post_decoder() -> decode.Decoder(Post) {
  use id <- decode.field(0, decode.int)
  use title <- decode.field(1, decode.string)
  use body <- decode.field(2, decode.string)
  use published <- decode.field(3, decode.bool)
  decode.success(Post(id: id, title: title, body: body, published: published))
}
```

The field order must match the `SELECT` column order.

## Migrations

Migrations are plain SQL files in `src/<app>/data/migrations/`.

### Generate a migration

```bash
refrakt gen migration add_email_to_posts
```

Creates: `src/<app>/data/migrations/002_add_email_to_posts.sql`

### Resource migrations (auto-generated)

`refrakt gen resource` creates the migration automatically.

PostgreSQL:

```sql
CREATE TABLE posts (
  id SERIAL PRIMARY KEY,
  title TEXT NOT NULL,
  body TEXT NOT NULL,
  published BOOLEAN NOT NULL,
  inserted_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

SQLite:

```sql
CREATE TABLE tasks (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  title TEXT NOT NULL,
  completed INTEGER NOT NULL,
  inserted_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### Running migrations

PostgreSQL:

```bash
psql my_app_dev < src/my_app/data/migrations/001_create_posts.sql
```

SQLite:

```bash
sqlite3 my_app.db < src/my_app/data/migrations/001_create_tasks.sql
```

### Programmatic migration runner

The `refrakt/migrate` module can run migrations from your app code:

```gleam
import refrakt/migrate

let count = migrate.run_from_directory(
  db_execute,      // fn(String) -> Result(Nil, String)
  db_query_strings, // fn(String) -> Result(List(String), String)
  "src/my_app/data/migrations",
)
```

The runner tracks applied migrations in a `_migrations` table and
only runs pending ones.

## Adding a database later

If you started without `--db` and want to add one:

1. Add the dependency: `gleam add pog` or `gleam add sqlight`
2. Create `src/<app>/data/repo.gleam` (copy from a generated project)
3. Add a `db` (or `db_path`) field to `Context` in `context.gleam`
4. Connect in `src/<app>.gleam` and pass to the context
5. Use `refrakt gen resource` — it auto-detects the DB from `gleam.toml`

## Writing queries manually

You own the repo files. Add custom queries alongside the generated ones:

```gleam
pub fn list_published(db: pog.Connection) -> List(Post) {
  pog.query("SELECT id, title, body, published FROM posts WHERE published = $1 ORDER BY id DESC")
  |> pog.parameter(pog.bool(True))
  |> pog.returning(post_decoder())
  |> pog.execute(db)
  |> result.map(fn(r) { r.rows })
  |> result.unwrap([])
}
```

There is no query builder or ORM. SQL is readable, debuggable, and
doesn't hide what's happening.
