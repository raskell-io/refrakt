# Project Structure

Every Refrakt project follows the same directory layout. There are no
configuration options for project structure — this is a convention, and
the generators depend on it.

## Directory layout

```
my_app/
  gleam.toml                            ← package config
  src/
    my_app.gleam                        ← entry point, starts the server

    my_app/
      config.gleam                      ← typed config, reads env vars
      context.gleam                     ← shared Context type (config + db)
      router.gleam                      ← all routes, pattern matched

      web/                              ← HTTP/UI layer
        home_handler.gleam              ← GET /
        error_handler.gleam             ← 404, 500 responses
        post_handler.gleam              ← generated CRUD handlers
        post_views.gleam                ← generated Lustre HTML views
        forms/
          post_form.gleam               ← generated form decoder + validation
        layouts/
          root_layout.gleam             ← HTML shell (head, body, CSS)
        components/
          flash.gleam                   ← flash message component
        middleware/
          auth.gleam                    ← auth middleware (if gen auth)
        islands/
          counter.gleam                 ← Lustre client app (if gen island)
          counter_embed.gleam           ← server-side embed helper

      domain/                           ← business logic
        post.gleam                      ← pure Gleam types, no framework imports
        user.gleam                      ← generated by gen auth
        auth.gleam                      ← password hashing (gen auth)

      data/                             ← persistence
        repo.gleam                      ← database connection setup
        post_repo.gleam                 ← SQL queries, typed decoders
        user_repo.gleam                 ← generated by gen auth
        migrations/
          001_create_posts.sql          ← plain SQL, run in order
          002_create_users.sql

  priv/
    static/
      css/app.css                       ← default stylesheet
      js/app.js                         ← client JavaScript

  test/
    my_app_test.gleam                   ← test entry point
    my_app/web/
      post_handler_test.gleam           ← generated tests
      auth_handler_test.gleam
```

## The three layers

### `web/` — HTTP and UI

Everything that touches HTTP lives here: handlers, views, forms,
layouts, components, middleware, islands.

- **Handlers** receive requests and return responses
- **Views** produce `Element(Nil)` using Lustre HTML functions
- **Forms** decode form data and validate input
- **Layouts** wrap view content in an HTML shell
- **Middleware** runs before handlers (auth, rate limiting)

### `domain/` — Business logic

Pure Gleam types and functions. **No framework imports allowed.**
No wisp, no lustre, no pog, no sqlight.

This is your code. Domain types like `Post`, `User`, and their
associated logic live here. They can be tested without any HTTP
or database setup.

### `data/` — Persistence

Database queries and migrations. Repo modules take a database
connection (or path) and return domain types.

- **Repos** use raw SQL with typed decoders (no ORM)
- **Migrations** are plain `.sql` files executed in order

## Dependency flow

```
web/ → domain/     ✓   handlers use domain types
data/ → domain/    ✓   repos return domain types
web/ → data/       ✓   handlers call repos

domain/ → web/     ✗   domain never imports HTTP concerns
domain/ → data/    ✗   domain never imports database concerns
data/ → web/       ✗   repos never import handlers
```

This keeps your business logic testable and portable.

## Key files

### `my_app.gleam` — Entry point

Starts the server. Loads config, connects to the database (if any),
creates the context, and starts Mist.

### `config.gleam` — Configuration

Reads environment variables: `PORT`, `SECRET_KEY_BASE`, `APP_ENV`.
Returns a typed `Config` record. No `.env` files — use your shell
or deployment tool.

### `context.gleam` — Shared context

The `Context` type holds everything handlers need: config and
database connection. It's passed to every handler as the second
argument.

### `router.gleam` — Route table

All routes in one file, using Gleam pattern matching:

```gleam
case wisp.path_segments(req), req.method {
  [], http.Get -> home_handler.index(req, ctx)
  ["posts"], http.Get -> post_handler.index(req, ctx)
  ["posts", id], http.Get -> post_handler.show(req, ctx, id)
  _, _ -> error_handler.not_found(req)
}
```

Generators patch this file by inserting routes before the catch-all.

### `root_layout.gleam` — HTML shell

Wraps view content in `<html>`, `<head>`, `<body>`. Includes the
CSS link and viewport meta. Every handler calls
`root_layout.wrap(content, title)`.
